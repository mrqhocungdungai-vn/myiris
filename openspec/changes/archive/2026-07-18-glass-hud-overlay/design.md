# Design — glass-hud-overlay

## Context

Upstream implementation (verified in `temp2/iris`): `HudShell.tsx` (270 lines; props include reactor/orb refs + keys, awake/caption, wake-word + mute toggles, tasks + steps toggles + scroll refs, transcript, hand-control state/stream/action labels, `onExitHud`) composes ReactorCore, WorkCard, HandSkeleton — all of which now exist in our tree post-ports. `hud.css` (468) styles it. Main-process side: overlay-capable BrowserWindow, `setIgnoreMouseEvents(true, {forward:true})` while the pointer is over glass and `(false)` over `.hud-hit` (renderer drives this via `hud:interactive`), `hud:toggle` handle, `hud:mode` emit for transitions, Tray with `updateTrayMenu()` (HUD toggle / wake / sleep), `globalShortcut.register(IRIS_HUD_HOTKEY || "Alt+Space")`, unregister on `will-quit`; tray icons `build/trayTemplate{,@2x}.png` generated by `scripts/render-icon.mjs`; electron-builder `files` includes them. Preload adds `toggleHud`, `setHudInteractive`, `onHudMode`, `windowControl` (`win:control`), `onWakeRequest`.

Our state after the three ports: renderer fully modular with Deep Space styles (hud.css deliberately not copied); `iris:sleep` and `iris:boot-done` exist (wake-sleep-voice); wake word arms while asleep; sounds/mute flag exist; WorkCard carries agent/model/⛓/StepTimeline; PO question banner is its own component; no Tray/globalShortcut/`screen` imports in `electron/main.mjs`; no `hud:*`, `iris:wake`, or `win:control` channels.

Claude-specific surfaces that upstream's HUD has no answer for: PO question banner (voice-decision-relay's click path), pipeline bar/model popover, SessionSwitcher, ProjectBar, SetupPanel.

## Goals / Non-Goals

**Goals:**
- Full upstream HUD parity: glass overlay, `.hud-hit` click-through islands, orb cluster, collapsible tasks column, comms, camera dock, deck↔HUD transitions.
- Three activation paths: UI toggle, global hotkey, tray; tray also offers wake/sleep.
- Claude parity inside the HUD: task cards render identically to the deck (agent badge, model, ⛓, StepTimeline), and a pending PO question is answerable in HUD mode (banner as `.hud-hit`).
- No regressions to deck mode, wake word, run queue, or the PO relay.

**Non-Goals:**
- No pipeline/role management, model selection, session switching, project attach, or SetupPanel inside the HUD — those remain deck-only; the HUD provides exit-to-deck.
- No Windows/Linux tray polish beyond not crashing (tray is macOS-first, matching upstream's `trayTemplate` approach); hotkey works everywhere.
- No new gesture vocabulary; HUD reuses the existing gesture engine (dwell-click works on `.hud-hit` controls because they are ordinary buttons).
- No multi-display placement logic beyond upstream's (primary display fullscreen overlay).

## Decisions

### D1 — Port upstream's click-through model unchanged
Renderer-driven interactivity: pointermove over `.hud-hit` → `setHudInteractive(true)` → main `setIgnoreMouseEvents(false)`; leaving → `(true, {forward:true})`. Rationale: OS-level hit-testing via `forward: true` is the only reliable Electron pattern for partial click-through, and upstream's is field-tested. Alternative (multiple small always-interactive windows) rejected: massive complexity, breaks shared React state and gesture overlay.

### D2 — PO question banner joins HudShell as a Claude-specific prop
`HudShell` gains a `poQuestion` slot rendering our `PoQuestionBanner` inside a `.hud-hit` container (positioned per hud.css conventions near the comms cluster). This is a deliberate, documented divergence from upstream's HudShell surface — kept as one additive prop so future upstream diffs stay clean. Voice answering needs nothing (it never touched the DOM); click/dwell answering requires the banner to be present and interactive. Suppression rule from voice-ui-control (TaskChooser hidden while a PO question pends) applies in HUD mode too.

### D3 — Management actions exit to deck rather than miniaturize
HUD keeps upstream's minimal surface (tasks, comms, orb, camera, mute/wake/sleep/exit). Role switching, model choice, sessions, project folder, and setup all require deck mode; the HUD's exit control (and `hud:toggle`/hotkey/tray) is the path back. Rationale: these are rare, deliberate actions; replicating them in the overlay bloats `.hud-hit` area and fights the ambient-glass concept. If a DEV soft-gate confirm (`window.confirm`) fires while in HUD, the native dialog appears above the overlay and works — acceptable, unchanged behavior.

### D4 — Tray, hotkey, and window flags are additive and isolated
`main.mjs` additions live in a clearly-fenced HUD/tray section (mirroring upstream's `// ===== Tray =====` block): `createTray()`, `updateTrayMenu()`, `hudHotkey()` (`IRIS_HUD_HOTKEY`, default `Alt+Space`), `globalShortcut` register-on-ready + unregister-on-will-quit with failure logged as a sidecar log event (never fatal — HUD stays reachable via UI toggle). Tray menu: toggle HUD, wake (`iris:wake`, new channel matching the existing keyboard wake path), sleep (reuses `iris:sleep`). None of this touches the Gemini bridge, run queue, or PO session code paths.

### D5 — `hud:mode` drives renderer transitions; mode is not persisted
Main is the source of truth for the current mode (it owns window state) and emits `hud:mode {mode}`; renderer animates `modeTransition` (deck-leaving/hud-entering per upstream classes). App always boots into deck mode; HUD is opt-in per run. Rationale: booting into a click-through overlay with no visible affordance is a lockout risk; upstream boots to deck as well.

### D6 — `win:control` adopted only if our TopBar already needs it
Upstream's preload exposes `windowControl` for TopBar window buttons. Our ported TopBar shipped without it in the restructure; include the channel only if the HUD/TopBar work actually wires a control that needs it (decided at implementation). Avoids dead IPC surface.

## Risks / Trade-offs

- [Click-through desync — window stays interactive over glass or inert over controls] → keep upstream's exact pointermove/leave wiring and `.hud-hit` discipline; smoke includes typing into an app underneath while HUD is up, and clicking every `.hud-hit` island.
- [Global hotkey collides with user/system shortcuts] → env-overridable (`IRIS_HUD_HOTKEY`), registration failure logs and degrades to UI/tray toggle.
- [PO question invisible if user is deep in another app with HUD glass up] → the banner is a lit `.hud-hit` island plus the existing `SYSTEM_EVENT_PO_QUESTION` voice announcement; timeout fallback (recommended option) still guards abandonment per the existing spec.
- [Gesture dwell-click in HUD hits tiny targets] → hud.css control sizing is upstream-tuned for dwell; `data-no-gesture` escape hatch from the gesture change remains available.
- [Tray icon assets missing in packaged builds] → electron-builder `files` delta is part of the change; packaged-mode smoke includes tray presence.
- [Fullscreen overlay vs macOS Spaces oddities] → mirror upstream flags exactly; note any Space-switching quirks in README known footguns rather than inventing new window management.

## Migration Plan

Additive feature; deck mode is untouched default. Land after verifying: `npm run build`, deck smoke unchanged, HUD smoke (below), packaged `.app` tray/hotkey check. Rollback = revert; no persisted state or data migration (mode never persists).

## Open Questions

- `win:control` inclusion (D6) — resolved at implementation when wiring TopBar/HUD controls.
- Exact HUD placement of the PO question banner (near comms vs above orb cluster) — visual call during implementation; requirement only fixes presence + interactivity.
