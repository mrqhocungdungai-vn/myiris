# Design: Bundled Pipeline Skills and Setup Guide

## Context

The chat-first release (archived change `community-release-chat-first`) left pipeline enablement as: install Claude CLI + openspec CLI, then hand-run two third-party install commands, then find the separate "Install agents" button. Investigation findings that shape this design:

- `REQUIRED_SKILLS` lists a `verify` skill that exists nowhere (not in mattpocock's set; the `iris-dev` persona invented it) — the SetupPanel check is permanently red even on the author's machine.
- `openspec init --tools claude` (v1.6.0) generates its 6 skills + 6 `/opsx` commands **project-locally** into `<cwd>/.claude/`, not user-level. The user-level copies on the author's machine are a legacy install. Checking user level for them fails on every fresh machine, yet user-level copies *do* work for any cwd — so the chosen fix is to make the installer populate user level, keeping the check where it is.
- The author's machine has `~/.claude/skills/*` entries that are **symlinks** managed by skills.sh (`~/.agents/skills/` + `.skill-lock.json`) — an overwriting installer would corrupt a tool-managed setup.
- Both skill sets are MIT (mattpocock/skills LICENSE; openspec SKILL.md `license: MIT`), so vendoring snapshots with attribution is legal.
- Skill payloads are tiny (8–20 KB per skill, markdown + an `agents/openai.yaml`), so bundling costs nothing.
- `installIrisAgents()` already exists (sync-overwrite policy, retired-agent cleanup) and stays the single implementation for persona installs.

Decisions locked with the user through grilling: bundle **both** sets from the repo (accepting snapshot lag vs upstream — maintainer refreshes manually), install-only-missing for third-party content, one "Install missing" action in SetupPanel that also installs agents, bilingual voice-first guide in `docs/`.

## Goals / Non-Goals

**Goals:**

- One explicit click takes a machine with the CLIs installed from red checks to a working pipeline: agents + skills + `/opsx` commands land in `~/.claude`.
- Checks tell the truth: no phantom `verify`, agents get their own row, the list matches exactly what the installer provides and the personas invoke.
- A community user can follow one guide (EN or VI) from zero to their first PO-grilled, DEV-implemented change — driven by voice, with the underlying `/opsx` flow explained, not assigned as homework.

**Non-Goals:**

- No auto-install without user action — the button is explicit; app startup never writes to `~/.claude`.
- No network at install time (snapshots only; `npx`/git fetches are out).
- No overwriting or updating of existing third-party skills — even outdated ones. Updating a skills.sh-managed install is skills.sh's job.
- No change to `ensureProjectScaffold` — user-level copies from the installer already cover the "cloned repo with existing `openspec/`" gap that project-local generation misses.
- No Windows/macOS installer packaging changes; this is all file copies inside the app.

## Decisions

### 1. Snapshot layout under `resources/skills/`

```
resources/skills/
  ATTRIBUTION.md            # sources, upstream commits/versions, MIT notices, refresh procedure
  claude-skills/            # → copied to ~/.claude/skills/<name>
    grilling/  tdd/  code-review/  diagnosing-bugs/     (from mattpocock/skills)
    openspec-propose/ … openspec-update-change/          (6, generated by openspec init 1.6.x)
  claude-commands/opsx/     # → copied to ~/.claude/commands/opsx/*.md (6 files)
```

Snapshots are plain copies of the real folders (symlink targets resolved). `ATTRIBUTION.md` records the upstream source, version/commit, and the refresh procedure (`npx skills add mattpocock/skills` on a dev machine + `openspec init` into a temp dir, then re-copy) — this is the maintenance contract the user accepted. Packaged builds resolve the directory like personas do: repo `resources/` in dev, `process.resourcesPath` when packaged (generalize `personasSourceDir()` into a shared resource-dir resolver rather than duplicating the probe).

### 2. One installer function, two write policies

`installPipelinePrereqs()` in `main.mjs` does, in order: (a) `installIrisAgents()` — unchanged sync-overwrite, because personas are Iris-owned and updates must propagate; (b) for each bundled skill directory and each bundled command file: if the destination exists (`lstat` — a symlink counts as existing), skip and report "already present"; otherwise copy recursively. The asymmetry is deliberate and documented in the guide: Iris owns the agents, third parties own the skills. Returns a structured report (`agents`, `installedSkills`, `skippedSkills`, `installedCommands`, `skippedCommands`, `errors`) so the panel can show what happened. Exposed over new IPC `pipeline:install-prereqs` + preload `installPipelinePrereqs()`; the PipelineBar's existing `agents:install` path is untouched.

### 3. Checks corrected to match what the installer provides

`REQUIRED_SKILLS` = `grilling`, `tdd`, `code-review`, `diagnosing-bugs`, `openspec-propose`, `openspec-apply-change`, `openspec-archive-change` (the exact set the personas invoke; `verify` dropped; the other three openspec skills install but aren't required). `checkClaudeHealth()` gains `agentsOk`/`missingAgents` (checks `~/.claude/agents/iris-po.md`, `iris-dev.md`). SetupPanel renders an agents row beside the skills rows; the "Install missing" button appears when any of agents/skills/commands are missing, runs the installer, then re-runs `checkClaude()` so rows flip green in place. Copyable manual commands stay as the fallback path for people who prefer their own tooling.

### 4. `verify` becomes an action, not a skill

`resources/personas/iris-dev.md` rewords "verify with the **`verify`** and **`code-review`** skills" to: switch hats and verify yourself — run every acceptance scenario for real, the typecheck, the full test suite, and the build; use the **`code-review`** skill for the review pass. `CLAUDE.md`'s pipeline sentence (`openspec-apply-change` + `tdd`/`verify`/`code-review`) drops `verify` the same way. No behavior change — Claude was already silently skipping the nonexistent skill.

### 5. Guide: voice-first, bilingual, one structure

`docs/PIPELINE_GUIDE.md` (EN) and `docs/PIPELINE_GUIDE.vi.md` (VI) share the same section skeleton so they can be diffed for drift: (1) what the pipeline is (one paragraph + the PO grill → propose → DEV apply → archive diagram); (2) setup — Claude CLI, `claude setup-token`, openspec CLI, then Settings → "Install missing"; (3) the voice walkthrough — what to say at each stage, what Iris says back, what the Work Stream shows, explicitly noting `/opsx:propose`/`/opsx:apply` are run by the agents, not typed by the user; (4) appendix — using `iris-po`/`iris-dev` and `/opsx:*` directly in Claude Code; (5) troubleshooting mapped to the SetupPanel rows. README's pipeline section shrinks to the enablement summary + a link to the guide. Cross-links at the top of both files ("Tiếng Việt →" / "English →").

## Risks / Trade-offs

- [Bundled snapshots lag upstream] → Accepted explicitly by the maintainer. `ATTRIBUTION.md` records versions + refresh steps; skills are prompt-ware, so staleness degrades quality, not safety. Never overwriting means a user can always install newer versions themselves and Iris won't fight them.
- [OpenSpec skill snapshot vs newer installed CLI] → The 1.6-generated skills mostly shell out to `openspec` subcommands that are stable; if a future CLI breaks them, the refresh procedure is the fix. Guide's troubleshooting notes this.
- [User has partial tool-managed installs (e.g. only `tdd` via skills.sh)] → Install-only-missing handles it per-entry: existing entries untouched, gaps filled from snapshots. Mixed provenance is acceptable; the report says which came from where.
- [Two guide languages drift] → Shared section skeleton + a maintenance note at the top of both files ("update both or mark the other stale"). Accepted as the cost of serving the Vietnamese community first-class.
- [Copying into `~/.claude` on Windows] → Plain file copies, no symlinks created — works on all three platforms.

## Migration Plan

1. Land snapshots + installer + checks + persona/docs edits in this change; archive syncs the spec deltas.
2. No data migration: everything is additive file copies gated behind an explicit button. Rollback = `git revert`; user machines keep whatever was installed (harmless standalone markdown).
3. After archive, the release notes for the next tag mention the one-click installer and the guide.

## Open Questions

- None blocking. Exact upstream commit hashes for `ATTRIBUTION.md` are captured at implementation time when the snapshots are produced.
