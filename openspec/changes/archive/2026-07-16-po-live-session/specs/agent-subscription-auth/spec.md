## ADDED Requirements

### Requirement: PO SDK session authenticates via subscription token

The PO SDK session SHALL authenticate using a subscription OAuth token supplied via `CLAUDE_CODE_OAUTH_TOKEN` (generated by `claude setup-token`), so that PO usage bills against the Claude subscription rather than the metered API. This requirement is scoped to the stateful PO module; the stateless DEV subprocess retains its existing `/login`-based subscription auth and is out of scope.

#### Scenario: PO SDK session uses the subscription token

- **WHEN** the app starts a live PO session and `CLAUDE_CODE_OAUTH_TOKEN` is set in the environment
- **THEN** the Agent SDK session authenticates with that token and runs against the subscription

#### Scenario: DEV auth is left unchanged

- **WHEN** the app spawns a DEV `claude -p` subprocess
- **THEN** it uses the existing DEV auth path unchanged and this change does not alter DEV's environment

#### Scenario: Missing token is reported clearly

- **WHEN** a PO session is attempted with no usable subscription token and no other configured credential
- **THEN** the app surfaces an actionable error naming `claude setup-token` / `CLAUDE_CODE_OAUTH_TOKEN` rather than failing silently

### Requirement: API key is excluded from the PO session environment

The app SHALL ensure `ANTHROPIC_API_KEY` is not present in the environment of the PO SDK session, so a stray API key cannot override the subscription token onto metered API billing. This env scrubbing is scoped to the PO module and does not modify the DEV subprocess environment.

#### Scenario: Stray API key in the host environment is stripped

- **WHEN** `ANTHROPIC_API_KEY` is present in the host/app environment and a PO session starts
- **THEN** the app removes `ANTHROPIC_API_KEY` from the environment passed to the PO session so the subscription token takes effect

#### Scenario: Billing path is subscription, not API

- **WHEN** both `CLAUDE_CODE_OAUTH_TOKEN` and (host) `ANTHROPIC_API_KEY` are available at launch
- **THEN** the PO session authenticates via the subscription token, because the API key was excluded from the PO session environment

### Requirement: Auth requirements are documented

The setup token requirement SHALL be documented in the app's configuration surface so an operator can enable subscription billing without reading the code.

#### Scenario: Configuration docs describe the token

- **WHEN** an operator consults `.env.example` / README to configure the app
- **THEN** they find the `claude setup-token` step, the `CLAUDE_CODE_OAUTH_TOKEN` variable, and the warning that `ANTHROPIC_API_KEY` forces API billing and is excluded from the PO session
