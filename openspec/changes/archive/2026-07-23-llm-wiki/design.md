## Context

Base Iris (Talk mode) already has a general worker — "plain Claude": `submit_claude_task` with no role runs `claude -p --resume` (see `startDevRun` in `electron/main.mjs`, the no-agent branch). It already remembers prior tasks (session resume) and has file/web tools, so it can already read and write markdown. What is missing is a *convention* — a defined vault and a skill that tells Claude how to keep an interlinked, retrievable knowledge base — plus the discoverability that turns "Claude can touch files" into "Iris is my second brain".

Claude Code discovers skills from `~/.claude/skills` globally; the DEV subprocess already relies on this to use `tdd`/`code-review`, so a skill dropped there is available to plain-Claude runs too. `installPipelinePrereqs()` already provisions `~/.claude/skills` by iterating `resources/skills/claude-skills/` and copying every missing directory. This change rides both mechanisms rather than inventing new ones.

## Goals / Non-Goals

**Goals:**
- A real personal-notes capability for Talk mode: capture + retrieve, in a user-owned Obsidian vault at `~/iris-second-brain`.
- Offline, zero extra runtime prerequisites (no `uv`/Python/ollama/API key).
- Reuse the existing installer and worker paths — minimal new code.
- Keep the vendored skills pristine (never edit snapshot content) so refresh stays a clean re-vendor.

**Non-Goals:**
- Upgrading quick tasks to a stateful Agent SDK session (explicitly rejected in grilling — resume is kept).
- Any new Gemini tool, UI, Google Search toggle, or role/two-mode guidance — those are the follow-up `clarify-role-capabilities` change.
- Semantic/embedding retrieval (the upstream skill's optional BM25/rerank extras are not adopted; plain read/grep retrieval only).

## Decisions

### D1: Vendor vanillaflava/llm-wiki-skills (6 skills), not author-from-scratch nor a heavier alternative
Chosen for being MIT and **filesystem-only** — no `uv`, Python, ollama, or API key — which is the single hard constraint for a skill that must install offline into `~/.claude/skills` and run under a headless `claude -p` worker with only file/web tools. Alternatives rejected: `AgriciDaniel/claude-obsidian` (15 skills, optional ollama/BM25/canvas/autoresearch — too heavy for a POC note-taker); `kfchou/wiki-skills` (needs `uv`/Python — a new runtime prerequisite); authoring our own (needless when a proven, right-sized MIT skill set exists). Bundle all 6 skills (`wiki-config`, `wiki-ingest`, `wiki-query`, `wiki-lint`, `wiki-integrate`, `wiki-crystallize`) — they are lightweight and cover capture, retrieval, and vault hygiene coherently.

### D2: Install via the existing `installPipelinePrereqs()` iterator — no installer code change
The installer already `readdirSync`s `resources/skills/claude-skills/` and copies each missing directory (copy-only-where-missing, symlink-safe). Dropping the 6 vendored dirs there is sufficient; the "Install missing" action provisions them with no loop change. **Not** added to `REQUIRED_SKILLS` / `checkSkillsStatus()`: that set is the PO/DEV pipeline prerequisite gate, and adding Talk-mode skills would falsely tell a chat-only user they have a missing prerequisite. The only spec impact is `pipeline-setup-install`'s enumeration/count, handled in its delta.

### D3: Pin the vault via `--append-system-prompt`, not by editing the skill
Plain-Claude runs execute with `cwd` = the workstream's project folder, so the LLM-Wiki root must be pinned explicitly or notes would scatter into whatever repo is open. Iris (a) ensures `~/iris-second-brain` exists before the run, and (b) appends a directive to `--append-system-prompt` that the personal-notes / LLM-Wiki vault root is `~/iris-second-brain` and the wiki skills operate there. This keeps the vendored snapshot untouched (honoring the ATTRIBUTION "do not edit content" rule) and is `cwd`-independent.

**Correction from initial draft:** `startDevRun` does not currently have a separate plain-Claude branch for `--append-system-prompt` — today it is built once as a single string shared unconditionally by plain-Claude and DEV runs, with role branching only happening later via the `--agent` flag push. So this decision requires *introducing* a conditional (`if (!run.agent)`) around the new vault-directive text, not just editing an already-isolated branch. PO/DEV runs must still end up unchanged — that is now an explicit verification step (task 2.2), not a given.

### D4: Vault at `~/iris-second-brain` (home root)
User-level so it is discoverable and Obsidian-openable via "Open folder as vault"; home root (not `~/Documents/...`) keeps the path short and predictable for the append-system-prompt directive and for the user to find. The skill's `wiki-config` scaffolds its config/templates on first use inside that folder; Iris only guarantees the directory exists.

### D5: Pre-seed `wiki-config.md` + `wiki-schema.md`, don't rely on the skill's own interactive Init flow
Discovered while vendoring (reading the actual `SKILL.md` files, not just the upstream README): `wiki-config`'s Init flow is built for a live back-and-forth — it asks the user "What's the path?" and waits for a reply before deploying anything. The 5 operational skills are stricter still: on a genuinely first-ever run (no `wiki-config.md` anywhere), their "Config Discovery" step finds nothing, does not consult any system-prompt directive, and ends the turn with *"Run `/wiki-config`... I'll wait for that to be done before proceeding."* Plain-Claude's run mode (`claude -p`, one-shot, no mid-run reply channel — unlike PO's voice-relayed `AskUserQuestion`) has nobody to answer that.

Decision: Iris pre-seeds `~/iris-second-brain/wiki-config.md` and `~/iris-second-brain/wiki-schema.md` itself (on the same "ensure vault exists" step as D4, idempotent — write only if absent, never touch if present) **before** the first plain-Claude run ever needs them, so the operational skills' Config Discovery always finds a valid config and never enters the interactive-ask branch. This is generated from wiki-config's own bundled `assets/wiki-config-template.md` / `assets/wiki-schema.md` (now vendored at `resources/skills/claude-skills/wiki-config/assets/`), not hand-authored — with one required deviation: the template's `blacklist` field ships as placeholder prose ("Folder(s) where the wiki should not write"), not real folder names, and `wiki-config`'s own Step-3 Validate flags placeholder text left in place. Since `~/iris-second-brain` is a single-purpose vault (nothing else lives there needing exclusion), Iris writes `blacklist: []` instead of copying the placeholder. `wiki-schema.md` is copied verbatim (no placeholders in it). Other Init-flow artifacts (`Home.md`, `Overview.md`, `index.md`, `log.md`, `raw/`, `ingested/`, `templates/`) are intentionally **not** pre-seeded — Config Discovery only gates on `wiki-config.md` + `wiki-schema.md`; the operational skills already have their own fallback behavior for a missing `templates/` folder ("falls back to a minimal hardcoded structure" per `wiki-config/SKILL.md`), so recreating the full 13-step Init in application code would be scope beyond what's needed to unblock first use.

This does not replace D3 — Claude still needs the append-system-prompt directive to know to look in `~/iris-second-brain` at all (the skills' recursive config search is scoped to the filesystem tool's reachable root, which for a project workstream is the project `cwd`, not the user's home directory).

### D6: The append-system-prompt directive checks skill installation, not just vault presence
Found during a follow-up walkthrough: vault creation (D4/D5, `ensureNotesVaultReady()`) and skill installation (`installPipelinePrereqs()`, the SetupPanel's "Install missing" button) are two independent actions on two independent schedules. `ensureNotesVaultReady()` runs automatically and silently on every plain-Claude run; installing the 6 `wiki-*` skills into `~/.claude/skills` happens only on that explicit button click. A user can therefore reach a state where `~/iris-second-brain` and its config already exist, but the wiki skills themselves are not installed — and the original D3/D5 directive ("use the wiki skills there for any note-taking request") would send Claude looking for skills that aren't there, with no defined fallback. Left unhandled, Claude would either invent an ungoverned ad-hoc note format or hallucinate the skill's behavior — neither is "no notes worker," the chat-only fallback the `personal-knowledge-notes` spec's "No Claude means no notes worker" scenario already promises for the Claude-CLI-absent case, but nothing previously covered the CLI-present-skills-absent case.

Decision: before building the vault directive, check `checkNotesSkillsStatus()` (all 6 `wiki-*` names present under `~/.claude/skills` — deliberately a separate check from `REQUIRED_SKILLS`/`checkSkillsStatus()`, which gates the PO/DEV pipeline only). If installed, append the existing D3/D5 directive. If not, append a different directive instead: tell Claude the notes skills aren't installed yet, and if the user asks to capture/retrieve a note, to say so and point at the SetupPanel's "Install missing" action rather than attempting anything ad hoc. This keeps the same one-branch structure as D3 (still gated on `!run.agent`, still leaves DEV/PO untouched) — it only changes what gets appended, not whether.

### D7: Surface notes-skill status as its own SetupPanel row, not folded into an existing check
Found from a screenshot during manual review: the SetupPanel's prerequisite list (openspec CLI / Global skills / Iris agents) had no row at all for the LLM-Wiki skills — a user had no visible way to tell whether the notes capability was actually installed, only the voice-level fallback from D6. This is a real, separate gap from D6: D6 is what Claude says if asked to use notes without the skills; D7 is whether the user can *see* the state at all without asking.

Decision: reuse `checkNotesSkillsStatus()` from D6 (same function, no new check invented) and thread it through the existing `checkClaudeHealth()` IPC response (`notesSkillsOk` / `missingNotesSkills` / `notesSkillsInstallHint`, parallel to the existing `skillsOk`/`agentsOk` fields) into a new `PrereqRow` in `SetupPanel.tsx`: **"Second-brain notes (LLM-Wiki skills)"**, alongside the existing three rows. Its missing state also joins the existing "show the Install missing button" condition — clicking it already installs the 6 `wiki-*` directories via the unchanged `installPipelinePrereqs()` loop (D2), so no new install path is needed, only visibility. This row is deliberately **informational, not a pipeline gate** — unlike `skillsOk`/`agentsOk`, `notesSkillsOk` being false never blocks PO/DEV (consistent with D2's original reasoning: this list is not `REQUIRED_SKILLS`).

## Risks / Trade-offs

- **[Append-system-prompt is a soft instruction — Claude could ignore it or write elsewhere]** → Word it imperatively and unambiguously, and pre-create the vault so there is one obvious target. Acceptable for a POC; a hard guarantee (e.g. sandboxing the run's writable root) is out of scope. As a cheap backstop (not a full guarantee), task 2.4 has Iris verify a matching file actually exists under `~/iris-second-brain` after a capture before confirming success to the user, so a misfire is reported as failed rather than silently claimed as saved.
- **[Upstream skill layout may not drop cleanly into `~/.claude/skills`]** (its documented install is `npx skills add` / Claude Desktop `.skill` upload) → At vendor time, verify each skill is a directory containing a valid `SKILL.md` and copy only those directories (mirroring how the other bundled skills are structured); confirm one real capture+retrieve round-trip before landing.
- **[Vault-scaffold expectations]** the skill may expect `raw/`/`templates/` folders → Let `wiki-config` create them on first use inside `~/iris-second-brain`; do not pre-seed beyond the directory itself unless testing shows it is required.
- **[Skill name collision in `~/.claude/skills`]** → The 6 `wiki-*` names do not collide with existing bundled skills; copy-only-where-missing also protects any user-managed entry.
- **[Vendored `SKILL.md` may hardcode its own vault/config convention]** (e.g. a default vault path or env var the upstream skill expects) that could conflict with the `~/iris-second-brain` pin → At vendor time (task 1.2), read each `SKILL.md` for any hardcoded path/config assumption and confirm the append-system-prompt directive actually overrides it in practice; do not edit the vendored file to force this — if a skill resists redirection, record it as a "Decision needed" rather than silently reinterpreting the requirement.
- **[Skill trigger phrasing may not match the spec's Vietnamese example utterances]** → The vendored skills' own trigger descriptions are authored in English, while the delta spec's scenarios use Vietnamese examples ("ghi chú lại", "tìm trong second-brain"). Confirmed during the end-to-end trial (task 4.5); if English-only triggers don't reliably fire on these phrases, that's a real gap to flag, not something to silently work around.

## Migration Plan

Additive and reversible. Rollback = remove the 6 vendored dirs from `resources/skills/claude-skills/` and revert the `startDevRun` append-system-prompt/ensure-dir edit; a user can delete `~/iris-second-brain` and the `~/.claude/skills/wiki-*` dirs to fully remove it. No data migration.

## Open Questions

- Exact upstream commit to pin for the ATTRIBUTION row — resolved at vendor time (latest `main` at vendoring, recorded with its SHA).
- Whether any of the 6 skills should be trimmed after a hands-on trial (e.g. if `wiki-integrate`/`wiki-lint` prove noisy for a casual note-taker) — default is bundle all 6; revisit only if the trial in the verify step shows friction. **Decision criteria if revisited:** trim a skill only if, during task 4's trial, it either (a) fires unprompted/unwanted on a plain capture or retrieve turn, or (b) errors out because a scaffold folder (`raw/`/`templates/`) it expects is missing and `wiki-config` doesn't create it. Absent either signal, keep all 6 — do not trim speculatively.
